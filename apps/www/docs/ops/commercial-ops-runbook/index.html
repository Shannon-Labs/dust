Commercial Ops Runbook

Dust is still in a manual beta-operations phase. The point of this runbook is to keep early revenue and support from depending on memory.

Invite Criteria

  • Clear team use case for local database workflows
  • Willingness to tolerate beta edges
  • Concrete evaluation timeline

Onboarding Steps

  1. Confirm fit and expectation setting.
  2. Send installation, docs, and sample-template links.
  3. Record the plan tier and any promised support level.
  4. Track follow-up owner and next check-in date.

Plan Changes

  • Upgrades and downgrades remain operator-managed until billing automation exists.
  • Record every plan change in a durable system, not private chat.

Support

  • Route product questions to the public support issue form first so the answer is reusable.
  • Route bugs through the bug template and docs problems through the docs feedback template.
  • Team beta and enterprise inquiries should start in the public intake forms, then move to direct follow-up only after the initial queue item exists.
  • Pull severe blockers into direct triage quickly.
  • Keep repeated questions feeding back into docs and FAQ updates.
  • One operator should own first response and next action so no intake thread becomes an orphan.

Refunds and Cancellations

  • Treat refunds as manual until the billing path is fully automated.
  • Record the reason for cancellation so packaging or onboarding gaps become visible.
